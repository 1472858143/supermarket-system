# 项目书补充说明（前端与架构增强）

## 1. 补充背景

现有项目书已经明确系统目标、后端模块、数据库基线和部署方案，但对前端结构、前端权限控制、现代 UI/UX、可维护性和可扩展性说明不够细。因此本补充说明作为项目书的增强章节，用于指导后续开发、测试、答辩和部署。

本补充说明不改变项目书的最高优先级，不新增项目书范围外业务模块。

## 2. 前端架构设计

前端采用 Vue 3 + Vite，整体结构如下：

```text
frontend
├─ src/api          接口封装
├─ src/components   公共组件
├─ src/layout       Header + Sidebar + Content 后台布局
├─ src/router       路由与访问控制
├─ src/stores       登录态、用户、角色状态
├─ src/utils        token 和权限工具
└─ src/views        业务页面
```

后台页面统一采用：

```text
AdminLayout
├─ Header：系统名称、当前用户、退出
├─ Sidebar：按角色过滤后的菜单
└─ Content：业务页面 RouterView
```

页面必须具备：

- 卡片式内容区
- 表格与分页
- 表单与基础校验
- 弹窗新增或编辑
- loading 状态
- empty 状态
- 操作成功和失败反馈

## 3. 权限系统设计

### 3.1 角色定义

| 角色编码 | 角色名称 | 定位 |
|---|---|---|
| `ADMIN` | 管理员 | 系统最高权限 |
| `USER` | 普通用户 | 基础业务操作用户 |

### 3.2 页面权限矩阵

| 页面 | ADMIN | USER |
|---|---:|---:|
| 首页概览 | 允许 | 允许 |
| 用户管理 | 允许 | 禁止 |
| 商品管理 | 允许 | 允许 |
| 库存管理 | 允许 | 允许 |
| 入库管理 | 允许 | 允许 |
| 出库管理 | 允许 | 允许 |
| 盘点管理 | 允许 | 允许 |
| 报表统计 | 允许 | 允许 |
| 系统信息 | 允许 | 允许 |

### 3.3 功能权限矩阵

| 功能 | ADMIN | USER |
|---|---:|---:|
| 用户查看、新增、编辑、删除、分配角色 | 允许 | 禁止 |
| 商品查看 | 允许 | 允许 |
| 商品新增、编辑、删除 | 允许 | 禁止 |
| 库存查看 | 允许 | 允许 |
| 库存上下限维护 | 允许 | 禁止 |
| 直接修改库存数量 | 禁止 | 禁止 |
| 入库查看、新增 | 允许 | 允许 |
| 出库查看、新增 | 允许 | 允许 |
| 盘点查看 | 允许 | 允许 |
| 盘点新增 | 允许 | 禁止 |
| 报表查看 | 允许 | 允许 |
| 报表写业务数据 | 禁止 | 禁止 |
| 系统信息查看 | 允许 | 允许 |
| 复杂系统配置 | 禁止 | 禁止 |

### 3.4 实现方式

- 路由权限：在路由 `meta.roles` 中声明角色。
- 菜单权限：根据当前用户角色过滤侧边栏。
- 按钮权限：通过 `PermissionButton` 控制操作按钮显示。
- 接口权限：后端通过认证拦截器解析 Token，并通过 `@RequireRoles` 校验角色。

## 4. 可维护性设计

后端遵循：

```text
Controller → Service → Domain/Mapper → Database
```

要求：

- Controller 不写业务逻辑。
- Service 是统一业务入口。
- Mapper 只做数据访问。
- DTO 接收请求参数。
- VO 返回前端展示数据。
- 不跨层调用。
- 入库、出库、盘点不能直接修改库存表。

前端遵循：

```text
View → API → request → Backend
View → Store
```

要求：

- 页面不直接写接口地址。
- API 文件只封装请求。
- Store 只管理登录态、用户、角色。
- 页面不写复杂业务规则。
- 页面不计算库存最终值。

## 5. 可扩展性设计

- 新增模块：新增后端业务包、前端 `views` 和 `api`，不修改库存核心规则。
- 新增角色：新增 `role_code` 和权限配置，不重构权限系统。
- 新增接口：保持 `/api` 前缀、统一响应、统一鉴权。
- 新增字段：DTO/VO 向后兼容。
- 新增库存变化来源：必须继续调用 `StockService`。
- 配置扩展：通过 Spring Profile、Vite 环境变量和系统环境变量实现。
- 密码算法升级：上层继续依赖 `PasswordService`，底层可由 BCrypt 替换为 Argon2id。

## 6. 安全设计

- 密码使用 BCrypt。
- 禁止明文密码入库。
- 登录响应、用户列表、用户详情均不返回 `password`。
- 日志不打印密码和密码哈希。
- Token 中包含用户 ID、用户名和角色编码。
- 后端接口必须做基础角色校验。
- 生产环境应使用 HTTPS 传输登录密码。

## 7. 部署设计

后端：

- `application.yml` 放公共配置。
- `application-dev.yml` 放开发环境配置。
- `application-prod.yml` 通过环境变量读取生产配置。

前端：

- `.env.development` 使用 `/api` 和 Vite proxy。
- `.env.production` 使用 `/api`，由 Nginx 反向代理。

禁止在代码中硬编码服务器 IP、数据库密码、JWT 密钥。
