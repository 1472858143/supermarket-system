# 超市库存管理系统容器化与自动部署设计

## 1. 目标与范围

本设计将现有 Vue 3、Spring Boot 3.3、MySQL 8 项目部署到华为云 Ubuntu 22.04 amd64 弹性云服务器。

当前部署用于本人测试和毕业答辩演示，通过服务器公网 IP 的 HTTP 80 端口访问。暂不启用域名、HTTPS 和公网长期运营。后续完成 ICP 备案后，可以在不改变应用容器结构的前提下接入域名和 HTTPS。

本阶段包括：

- 前端、后端和数据库容器化。
- 使用 Docker Compose 编排单机服务。
- 从空数据库自动初始化完整表结构。
- GitHub `main` 分支推送后，通过 GitHub Actions 自动部署。
- 提供服务器初始化、密钥配置、部署、备份和故障排查说明。

本阶段不包括：

- Kubernetes、多节点高可用和自动扩缩容。
- 公网域名、HTTPS、CDN、WAF 和 ICP 备案实施。
- 本地历史数据迁移。
- 独立镜像仓库和镜像版本发布流程。

## 2. 已确认约束

- 服务器：华为云 ECS，Ubuntu 22.04 Server 64 位，amd64。
- 资源：2 核 CPU、4 GB 内存、40 GB 系统盘。
- 仓库：GitHub 公开仓库，默认分支为 `main`。
- 访问范围：本人测试和毕业答辩演示。
- 数据库：首次部署使用全新空库。
- 发布方式：GitHub Actions 通过 SSH 连接服务器并执行部署。
- 对外端口：仅开放 SSH 端口和 HTTP 80；后端 8080、MySQL 3306 不暴露公网。

## 3. 总体架构

系统使用一个 Docker Compose 项目运行三个服务：

```text
Internet
   |
   | HTTP :80
   v
frontend (Nginx + Vue dist)
   |-- /             -> Vue SPA 静态资源
   `-- /api/*        -> backend:8080
                              |
                              v
                         mysql:3306
                              |
                              v
                    Docker named volume
```

容器之间通过 Compose 私有网络使用服务名通信。只有前端容器映射宿主机 80 端口。后端和数据库仅使用容器内部端口。

## 4. 容器设计

### 4.1 前端容器

前端采用多阶段镜像：

1. Node.js 构建阶段执行 `npm ci` 和 `npm run build`。
2. Nginx 运行阶段仅复制 `dist` 静态文件和生产配置。

Nginx 负责：

- 使用 `try_files` 将未知前端路由回退至 `index.html`，支持 Vue Router history 模式。
- 将 `/api/` 请求代理至 `backend:8080`。
- 传递客户端地址、Host 和协议等必要请求头。
- 提供前端容器健康检查入口。
- 对静态资源设置合理缓存，对 `index.html` 禁止长期缓存。

前端生产构建继续使用相对 API 前缀 `/api`，因此从公网 IP 切换到域名时不需要重新修改前端接口地址。

### 4.2 后端容器

后端采用多阶段镜像：

1. Maven + JDK 21 构建阶段执行测试和 Jar 打包。
2. JRE 21 运行阶段仅复制可执行 Jar，并以非 root 用户启动。

后端使用 `prod` Profile，通过环境变量读取数据库地址、账号、密码、JWT 密钥和启动账号配置。数据库地址固定使用 Compose 服务名：

```text
jdbc:mysql://mysql:3306/market
```

Java 运行内存设置需适配 4 GB 服务器，避免与 MySQL、Docker 构建争抢内存。初始建议限制后端容器内存并设置约 512 MB 的 JVM 最大堆；实际值在实施时通过压力较小的演示场景验证。

后端需要提供可用于容器探测的轻量健康检查。若不引入 Spring Boot Actuator，则使用现有无副作用接口或新增最小健康端点；健康端点不能访问或修改业务数据。

### 4.3 MySQL 容器

数据库使用 MySQL 8 官方镜像，并配置：

- 独立命名卷持久化 `/var/lib/mysql`。
- UTF-8 MB4 字符集和 `Asia/Shanghai` 时区。
- 数据库名、应用账号和密码通过环境变量注入。
- 首次创建数据卷时，通过 `/docker-entrypoint-initdb.d` 执行完整初始化脚本。
- 健康检查通过 `mysqladmin ping` 完成。

数据库初始化必须只采用一个经过确认的完整快照脚本，不能同时执行全量脚本和增量迁移脚本。实施前需验证 `system/sql/00_终版.sql` 已包含当前代码需要的全部表和字段；若不完整，应生成新的容器初始化快照，而不是在 Compose 中串联历史脚本。

MySQL 不映射宿主机 3306 端口。需要维护时通过 `docker compose exec mysql` 进入容器操作。

## 5. Compose 编排

`compose.yaml` 定义 `frontend`、`backend`、`mysql` 三个服务以及数据库命名卷。

启动顺序与就绪条件：

1. MySQL 启动并通过健康检查。
2. 后端在 MySQL 健康后启动，并通过自身健康检查。
3. 前端在后端启动后运行。

所有服务设置 `restart: unless-stopped`。容器日志配置轮转上限，防止 40 GB 系统盘被日志持续占满。构建上下文和 `.dockerignore` 排除 `node_modules`、`target`、`dist`、Git 元数据、论文文档及其他无关大文件。

生产秘密不写入 Git。仓库只提供 `.env.example`，服务器保存实际 `.env`，至少包含：

- `MYSQL_ROOT_PASSWORD`
- `DB_USERNAME`
- `DB_PASSWORD`
- `JWT_SECRET`
- `BOOTSTRAP_ADMIN_USERNAME`
- `BOOTSTRAP_ADMIN_PASSWORD`
- `BOOTSTRAP_USER_USERNAME`
- `BOOTSTRAP_USER_PASSWORD`

演示账号初始化可在首次部署时开启。初始化成功后，应更换默认密码；后续部署不依赖重复初始化账号。

## 6. 自动部署流程

GitHub Actions 在 `main` 分支推送时触发，也支持手动触发。工作流包含：

1. 检出代码。
2. 执行后端测试。
3. 执行前端生产构建。
4. 通过 SSH 登录服务器。
5. 在固定部署目录更新公开仓库代码。
6. 执行 `docker compose up -d --build --remove-orphans`。
7. 等待健康检查并验证本机 HTTP 响应。
8. 清理无引用的构建缓存和旧镜像，但不删除运行镜像和数据库卷。

GitHub 仓库 Secrets 至少保存：

- `DEPLOY_HOST`：服务器公网 IP。
- `DEPLOY_PORT`：SSH 端口。
- `DEPLOY_USER`：专用部署用户。
- `DEPLOY_SSH_KEY`：部署私钥。

服务器上的 `.env` 不通过 GitHub Actions 覆盖，以减少生产秘密传输和误删风险。部署用户加入 `docker` 组，仅获得部署所需权限，不使用 root 密码登录。SSH 优先使用密钥认证，并在验证成功后关闭密码登录。

若任何测试、构建、容器启动或健康检查失败，工作流失败并保留日志。当前阶段不实现自动数据库回滚；数据库结构变更必须先备份并采用向后兼容迁移。

## 7. 服务器与网络安全

服务器初始化包括：

- 更新系统安全补丁。
- 安装 Docker Engine 和 Compose Plugin。
- 创建专用部署目录和部署用户。
- 配置 SSH 公钥认证。
- 配置华为云安全组及 Ubuntu 防火墙。

公网入站规则：

- SSH：仅允许管理员固定 IP 时优先限制来源；若网络地址不固定，至少使用密钥认证并考虑修改默认端口。
- HTTP 80：允许演示访问。
- 8080、3306：禁止公网入站。

由于当前使用裸 IP，不配置 HTTPS。系统只用于低敏感度演示，不应录入真实员工、供应商、库存或密码数据。备案和域名准备完成后，再增加 443、TLS 证书和 HTTP 到 HTTPS 跳转。

## 8. 数据持久化与备份

数据库数据存储在 Docker 命名卷中，重新构建或替换容器不会删除数据。任何部署和清理命令不得使用 `docker compose down -v`。

提供手动备份命令，将 `mysqldump` 输出保存到宿主机受限目录。至少在以下操作前备份：

- 执行数据库结构迁移。
- 升级 MySQL 主版本。
- 修改数据库卷或 Compose 项目名。
- 进行可能影响数据的维护操作。

40 GB 磁盘需定期检查 Docker 占用、数据库卷、部署目录和备份文件。备份保留策略以演示环境为主，避免无限累积。

## 9. 错误处理与恢复

- MySQL 未就绪：后端不启动或由重启策略重试，前端仍可返回静态页面但 API 不可用。
- 后端启动失败：健康检查失败，部署工作流返回失败；通过后端容器日志定位原因。
- 前端代理失败：Nginx 返回网关错误，不直接暴露后端端口。
- 初始化脚本失败：删除空的首次初始化卷后修正脚本并重新初始化；仅允许在确认卷中没有有效数据时执行。
- 新版本异常：代码可回退到上一个 Git 提交后重新构建；数据库变更不可假设可自动回滚。
- 磁盘不足：日志轮转和受控镜像清理降低风险，数据库卷和备份不得被自动清理。

## 10. 测试与验收

本地验证：

- 后端测试通过。
- 前端生产构建通过。
- Docker 镜像构建成功。
- 在全新数据卷上完成数据库初始化。
- 三个服务健康检查通过。

服务器验收：

- `http://公网IP` 能打开登录页，刷新任意前端路由不返回 404。
- `/api` 请求经 Nginx 转发，公网无法直接访问 8080 和 3306。
- 管理员和普通用户可以使用新设置的演示账号登录。
- 核心入库、出库、盘点和报表流程可用。
- 重启服务器后容器自动恢复，数据库数据仍存在。
- 推送 `main` 后 GitHub Actions 能完成自动部署。
- 部署失败时工作流明确失败，不显示虚假成功。
- 容器日志存在大小和文件数量上限。

## 11. 后续域名升级

完成 ICP 备案后：

1. 在阿里云 DNS 将域名解析到华为云 ECS 公网 IP。
2. 在入口 Nginx 增加域名配置和 TLS 证书。
3. 开放 443，并将 80 重定向至 HTTPS。
4. 在网页底部展示 ICP 备案号并链接工信部备案系统。
5. 网站开通后按要求办理公安联网备案并展示公安备案号。

前端保持 `/api` 相对路径，后端和 MySQL 网络结构无需调整。
