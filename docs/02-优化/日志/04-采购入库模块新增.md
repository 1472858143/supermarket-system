# 04 - 采购入库模块新增

> 记录时间：2026-06-01
> 项目阶段：优化阶段 — 新增独立采购入库模块，作为日常入库主入口

---

## 一、背景与动机

原系统仅有一个通用入库模块（inbound_order），存在以下不足：

- 无法区分采购入库与其他入库（期初入库、赠品入库、手动补录）
- 单条记录结构，无主表/明细拆分，不支持一次入库多个 SKU
- 无采购金额、成本单价计算
- 库存流水统一为 INBOUND，无法按入库来源分析

本次新增独立的采购入库模块，支持主表+明细结构、多 SKU 批量入库、自动成本核算，并通过 StockService 统一完成库存变化。

---

## 二、设计决策

| 决策点 | 方案 | 理由 |
|---|---|---|
| 仓库概念 | 暂不引入 | 当前单仓库系统，YAGNI |
| StockService 改动 | 修改 increase() 签名加 changeType 参数 | 只有 1 处现有调用方，改动干净 |
| 库存流水类型 | 新增 PURCHASE_INBOUND | 区分采购入库与普通入库 |
| 普通入库模块 | 保留不动 | 后续用于期初入库、赠品入库、手动补录 |
| 单据状态 | 创建即完成，status = COMPLETED | 当前无审批流，YAGNI |
| 供应商 | 不引入 | 后续再扩展 |
| 单号格式 | PI{YYYYMMDD}{NNN} | 简洁、可排序、含日期 |

## 六、业务规则汇总

| 规则 | 说明 |
|---|---|
| 创建即入库 | 保存单据 + 增加库存在同一事务中 |
| 库存流水类型 | PURCHASE_INBOUND，区别于普通入库 INBOUND |
| 成本单价计算 | cost_price = purchase_price / conversion_rate（4 位小数，HALF_UP） |
| 小计金额计算 | amount = quantity × purchase_price |
| 基础数量换算 | base_quantity = quantity × conversion_rate |
| 单号自动生成 | PI + 日期 + 三位递增序号 |
| 单位解析 | 复用 SkuUnitResolver，校验 SKU 存在 + 单位有效 |
| 操作人 | 从当前登录用户自动获取 |
| 事务一致性 | 单据保存失败则库存不增；库存增加失败则单据回滚 |

## 七、stock_log change_type 值域
