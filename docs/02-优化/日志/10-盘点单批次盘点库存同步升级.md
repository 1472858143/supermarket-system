# 10 - 盘点单批次盘点库存同步升级

> 记录时间：2026-06-02
> 项目阶段：优化阶段 - 将盘点从 SKU 直接调整升级为盘点单驱动的批次级库存调整

---

## 一、背景与目标

前序优化已经完成采购入库生成库存批次、出库按批次同步扣减、批次生命周期管理等能力。盘点模块仍停留在旧模式：选择 SKU 后直接把 `stock.quantity` 调整为实盘数量。

这种模式存在几个问题：

- 盘点只影响 SKU 总库存，不能说明差异来自哪个批次。
- `stock.quantity` 与 `stock_batch.quantity` 可能被盘点操作打散。
- 盘点只写 `stock_log`，缺少 `stock_batch_log`，批次账无法闭环追溯。
- 盘点没有正式单据状态，无法支持“创建范围、逐批录入、统一完成”的业务流程。

本次升级目标：

- 盘点单支持名称和自动编号。
- 支持按全部商品、一级分类、二级分类、指定 SKU 创建盘点范围。
- 指定 SKU 支持全部、多选、单选。
- 系统根据 SKU 自动展开可盘点批次。
- 盘点人员录入每个批次的实际数量。
- 提交后按差异同步调整批次库存和 SKU 总库存。
- 同时写入 `stock_log` 和 `stock_batch_log`。
- 盘点完成后状态变为 `COMPLETED`。

---

## 二、核心设计

盘点从“即时调整记录”升级为“盘点单 + 批次明细”：

```text
创建盘点单
  ↓
解析盘点范围内 SKU
  ↓
自动展开 stock_batch 为 stock_check_item
  ↓
盘点人员录入每个批次 actual_quantity
  ↓
完成盘点
  ├─ 锁定盘点单
  ├─ 锁定 SKU 总库存
  ├─ 锁定批次库存
  ├─ 更新 stock_batch.quantity
  ├─ 汇总同步 stock.quantity
  ├─ 写 stock_batch_log
  ├─ 写 stock_log
  └─ stock_check.status = COMPLETED
```

关键口径：

| 项目 | 规则 |
|---|---|
| 盘点调整粒度 | 必须落到具体库存批次 |
| SKU 总库存 | 不再直接由盘点录入，而是由批次库存汇总同步 |
| 批次差异 | `提交时实盘数量 - 提交时批次当前数量` |
| 明细快照 | `system_quantity` 保留创建盘点单时的批次账面数量 |
| 日志闭环 | 完成盘点同时写 `stock_log` 和 `stock_batch_log` |
| 完成状态 | 成功提交后盘点单状态变为 `COMPLETED` |

## 三、数据库变更

新增迁移脚本：

```text
system/sql/12_upgrade_stock_check_order.sql
```

### `stock_check`

旧的 `stock_check` 是 SKU 盘点记录表。本次迁移会将旧表归档为：

```text
stock_check_legacy
```

然后重新创建 `stock_check` 作为盘点单头：

| 字段 | 说明 |
|---|---|
| `check_no` | 自动编号，格式 `PCyyyyMMddNNNN` |
| `name` | 盘点名称 |
| `scope_type` | `ALL` / `CATEGORY_LEVEL1` / `CATEGORY_LEVEL2` / `SKU` |
| `category_id` | 分类范围时使用 |
| `sku_select_type` | `ALL` / `MULTI` / `SINGLE` |
| `status` | `DRAFT` / `COMPLETED` / `CANCELLED` |
| `total_sku_count` | 本次展开的 SKU 数 |
| `total_batch_count` | 本次展开的批次数 |
| `total_difference` | 完成盘点后的总差异 |
| `create_time` | 创建时间 |
| `complete_time` | 完成时间 |

### `stock_check_item`

新增盘点批次明细表：

| 字段 | 说明 |
|---|---|
| `stock_check_id` | 所属盘点单 |
| `sku_id` | SKU |
| `stock_batch_id` | 库存批次 |
| `batch_no` | 批次号快照 |
| `system_quantity` | 创建盘点单时的批次账面数量 |
| `actual_quantity` | 盘点录入的实际数量 |
| `difference` | `actual_quantity - system_quantity`，用于盘点报表展示 |
| `status` | `PENDING` / `COUNTED` |
| `expire_date` | 到期日期快照 |

### 批次库存约束调整

原 `stock_batch` 存在：

```sql
CHECK (quantity <= initial_quantity)
```

本次移除该限制。原因是盘点可能出现盘盈，实盘数量可能超过原始入库数量；继续保留该约束会导致批次盘盈无法提交。

```text

```

旧 DTO 只支持 `skuId + actualQuantity`，已经不符合批次级盘点模型。

### `StockCheckController`

接口升级：

| 方法 | 路径 | 说明 |
|---|---|---|
| `GET` | `/api/stockchecks` | 查询盘点单列表 |
| `POST` | `/api/stockchecks` | 创建盘点单并展开批次 |
| `GET` | `/api/stockchecks/{id}` | 查看盘点单详情和批次明细 |
| `PUT` | `/api/stockchecks/{id}/items` | 保存批次实盘数量 |
| `POST` | `/api/stockchecks/{id}/complete` | 完成盘点并同步库存 |

### `StockCheckService`

核心能力：

| 方法 | 说明 |
|---|---|
| `create(...)` | 校验范围，解析 SKU，展开可盘点批次，写入 `stock_check` 和 `stock_check_item` |
| `updateItems(...)` | 仅允许 `DRAFT` 状态保存实盘数量，更新明细为 `COUNTED` |
| `complete(...)` | 锁单、锁库存、锁批次，更新库存并写日志，最后状态改为 `COMPLETED` |

完成盘点时的关键处理：

- 盘点单必须为 `DRAFT`。
- 所有明细必须已录入实际数量。
- 按 SKU 升序分组处理，降低并发提交时锁顺序不稳定带来的死锁风险。
- 每个 SKU 先锁定 `stock`。
- 每个批次使用 `findByIdAndSkuIdForUpdate(...)` 锁定。
- 批次更新后写 `stock_batch_log`：

```text
change_type = CHECK
source_type = STOCK_CHECK
source_id = stock_check.id
```

- SKU 总库存使用 `stock_batch` 汇总值同步，而不是简单累加差异：

```text
stock.quantity = sum(stock_batch.quantity where sku_id = ?)
```

- 同步后写 `stock_log`：

```text
change_type = CHECK
change_quantity = stockAfterQuantity - stockBeforeQuantity
```

### `StockBatchMapper`

新增：

| 方法 | 说明 |
|---|---|
| `findCheckableBySkuIds(...)` | 根据 SKU 列表查询可盘点批次，排除 `CLOSED` |
| `sumQuantityBySkuId(...)` | 汇总某 SKU 所有批次数量，用于同步 SKU 总库存 |

## 七、影响与注意事项

### 旧接口影响

旧的按 SKU 直接提交盘点：

```json
{
  "skuId": 1,
  "actualQuantity": 10
}
```

已经不再作为盘点入口使用。盘点必须先创建盘点单，再录入批次实盘数量并完成。

### 数据库迁移注意

迁移脚本会执行：

```sql
RENAME TABLE stock_check TO stock_check_legacy;
```

因此执行前需要确认：

- 当前库中不存在同名 `stock_check_legacy`。
- 旧盘点记录需要保留时，可从 `stock_check_legacy` 查询。
- 新代码依赖新的 `stock_check` 和 `stock_check_item` 结构。

### 库存一致性口径

完成盘点后：

```text
stock.quantity = 对应 SKU 的全部 stock_batch.quantity 汇总
```

这意味着 `stock` 是 SKU 汇总缓存，批次库存才是盘点调整的事实来源。

## 八、后续优化方向

- 盘点单支持取消 `CANCELLED` 流程。
- 盘点单支持按未盘、已盘、差异批次筛选。
- 盘点完成前支持导出或打印盘点表。
- 盘点完成后提供差异报表。
- 支持多人盘点分工和盘点人字段。
- 对盘点期间发生出入库的批次提供差异提示。

---

*本次升级后，盘点链路从“直接调整 SKU 总库存”升级为“盘点单驱动的批次级调整”。库存调整、SKU 汇总库存、SKU 总账流水和批次流水在同一事务中闭合，库存账务边界更加清晰。*
