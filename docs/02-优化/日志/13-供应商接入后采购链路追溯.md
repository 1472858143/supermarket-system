# 13 - 供应商接入后采购链路追溯

> 记录时间：2026-06-02
> 项目阶段：优化阶段 — 对供应商接入后的采购全链路进行追溯梳理，识别数据流转关系与潜在断点

---

## 一、追溯目的

供应商基础管理（`05`）与供应商SKU绑定（`12`）已落地，采购入库已强制关联供应商。本次追溯目标：

- 梳理从供应商建立到库存变动的完整数据链路
- 确认各模块间的引用关系与校验覆盖
- 识别链路中的断点与可改进项

本次为纯梳理，不涉及代码改动。

---

## 二、链路全景

```
┌──────────┐    绑定     ┌────────────┐    创建      ┌──────────────────┐
│ Supplier │───1:N──────▶│SupplierSku │◀──校验───────│PurchaseInbound   │
└──────────┘             └─────┬──────┘              └────────┬─────────┘
                               │ sku_id                       │ 1:N
                               ▼                              ▼
                          ┌─────────┐              ┌────────────────────┐
                          │   Sku   │              │PurchaseInboundItem │
                          └────┬────┘              └─────────┬──────────┘
                               │ sku_id                      │ 1:1
                               ▼                             ▼
                          ┌─────────┐              ┌──────────────────┐
                          │  Stock  │◀──汇总更新───│   StockBatch     │
                          └─────────┘              └──────────────────┘
```

正向追溯（供应商 → 库存）：Supplier → supplier_sku → purchase_inbound → purchase_inbound_item → stock_batch → stock

反向追溯（库存 → 供应商）：stock.sku_id → stock_batch.purchase_inbound_item_id → purchase_inbound_item → purchase_inbound.supplier_id → supplier

## 三、各环节追溯明细

### 3.1 供应商建立

| 项目 | 内容 |
|---|---|
| 表 | `supplier` |
| 关键字段 | `supplier_code`, `supplier_name`, `status` |
| 操作入口 | `POST /api/suppliers` |
| 业务约束 | 唯一编码，status=1 启用 / status=0 停用 |

### 3.2 供应商 × SKU 供货绑定

| 项目 | 内容 |
|---|---|
| 表 | `supplier_sku` |
| 关键字段 | `supplier_id`, `sku_id`, `supplier_sku_code`, `default_purchase_price`, `min_purchase_quantity`, `status` |
| 唯一约束 | `UNIQUE KEY (supplier_id, sku_id)` — 同一供应商对同一 SKU 仅一条绑定 |
| 操作入口 | `POST /api/suppliers/{id}/skus` |
| 删除保护 | 若该 (supplier_id, sku_id) 已被 `purchase_inbound_item` 引用，则禁止删除，只能禁用 |
| 前置校验 | 供应商存在、SKU 存在、不重复绑定 |

### 3.3 采购入库单创建

| 项目 | 内容 |
|---|---|
| 主表 | `purchase_inbound` |
| 明细表 | `purchase_inbound_item` |
| 单号格式 | `PI{YYYYMMDD}{NNN}`（当日自增） |
| 操作入口 | `POST /api/purchase-inbounds` |
| 操作人 | 从 `CurrentUserContext` 获取当前登录用户名 |

创建时的校验链（按执行顺序）：

```
① requireEnabledSupplier(supplierId)            — 供应商存在且启用
② 对每个明细项：
   ├─ validateBatchFields()                      — 生产日期必填，保质期>0
   ├─ supplierSkuService.requireEnabledBinding() — 绑定存在且启用
   ├─ validateMinPurchaseQuantity()              — 采购量 ≥ 最小采购量
   ├─ skuUnitResolver.resolve()                  — 解析单位换算率
   ├─ calculateBaseQuantity()                    — 基础数量 = 数量 × 换算率
   ├─ validatePurchasePrice()                    — 采购价 ≤ 1亿，最多2位小数
   └─ validateAmount()                           — 小计金额 ≤ 100亿
③ 生成单号 → insertInbound → 逐条 insertItem
④ 逐条调用 stockService.increase()
```

### 3.4 库存增加（Stock 层）

| 项目 | 内容 |
|---|---|
| 方法 | `StockService.increase(StockIncreaseCommand)` |
| 并发控制 | `SELECT ... FOR UPDATE` 行锁 |
| 汇总表变更 | `UPDATE stock SET quantity = quantity + baseQuantity` |
| 流水记录 | `INSERT stock_log`，change_type = `PURCHASE_INBOUND` |
| 成本价计算 | `costPrice = purchasePrice ÷ conversionRate`（4位精度） |

### 3.5 库存批次创建（StockBatch 层）

| 项目 | 内容 |
|---|---|
| 方法 | `StockBatchService.createFromPurchaseInboundItem()` |
| 批次号格式 | `SB{YYYYMMDD}{NNN}`（当日自增，上限999） |
| 初始状态 | `AVAILABLE` |
| 关键约束 | `purchase_inbound_item_id` 为 UNIQUE KEY，一个明细仅产生一个批次 |
| 到期计算 | `expire_date = production_date + shelf_life_days` |
| 流水记录 | `INSERT stock_batch_log`，change_type = `PURCHASE_INBOUND` |

## 五、批次生命周期

采购入库创建的批次后续经历以下状态变迁：

```
采购入库 ──▶ AVAILABLE ──┬──▶ DEPLETED   (出库耗尽)
                         ├──▶ EXPIRED    (过期扫描)
                         ├──▶ LOCKED     (手动锁定)
                         ├──▶ DAMAGED    (报损)
                         └──▶ CLOSED     (归档关闭，quantity=0)
```

| 变更路径 | 触发方式 | 对 stock 汇总表的影响 |
|---|---|---|
| → AVAILABLE | 采购入库自动创建 | quantity 增加 |
| AVAILABLE → DEPLETED | 出库时 FEFO 消费 | quantity 减少 |
| AVAILABLE → EXPIRED | 过期扫描定时任务 | 无直接影响 |
| AVAILABLE/EXPIRED → LOCKED | 手动锁定 | 无直接影响（锁定批次不参与出库） |
| AVAILABLE/EXPIRED/LOCKED → DAMAGED | 手动报损 | quantity 减少 |
| quantity=0 → CLOSED | 手动归档 | 无影响 |

出库消费规则（FEFO）：按 `expire_date ASC, id ASC` 排序，优先消耗最早到期的批次。

## 六、数据痕迹示例

以一次采购 2 箱可乐（供应商 A，SKU=1001，每箱24瓶）为例，入库后数据库中产生的记录：

```
supplier (供应商A)
  └─ supplier_sku: supplier_id=A, sku_id=1001, default_purchase_price=48.00, min_purchase_quantity=1

purchase_inbound: id=1, order_no=PI20260602001, supplier_id=A, total_quantity=48, total_amount=96.00
  └─ purchase_inbound_item: id=1, sku_id=1001, quantity=2, unit="箱", conversion_rate=24,
                            base_quantity=48, purchase_price=48.00, cost_price=2.0000, amount=96.00

stock: sku_id=1001, quantity=48（假设此前为0）
  └─ stock_log: change_type=PURCHASE_INBOUND, change_quantity=48, before=0, after=48

stock_batch: id=1, batch_no=SB20260602001, sku_id=1001, purchase_inbound_item_id=1,
             initial_quantity=48, quantity=48, status=AVAILABLE, purchase_price=48.00,
             production_date=2026-05-20, shelf_life_days=365, expire_date=2027-05-20
  └─ stock_batch_log: change_type=PURCHASE_INBOUND, change_quantity=48, before=0, after=48
```

## 七、已识别的链路断点与观察

| # | 观察点 | 现状 | 影响 |
|---|---|---|---|
| 1 | 采购单无草稿/审批流 | 创建即 COMPLETED | 无法追溯"谁提交了但未确认"的采购意图 |
| 2 | operator 仅记录用户名 | 取自 CurrentUserContext.getUsername() | 用户改名后历史记录无法精确关联到人 |
| 3 | 供应商侧商品名未落库到 item | supplier_sku_name 仅用于前端展示 | 从入库明细无法回溯"供应商侧叫什么名字" |
| 4 | 一个 item 只产生一个 batch | purchase_inbound_item_id 是 UNIQUE KEY | 同一 SKU 不同生产日期需拆成多条明细 |
| 5 | 成本价换算精度 | costPrice = purchasePrice / conversionRate，conversionRate 为 int | 换算率非整除时存在精度累积（如 10÷3=3.3333） |
| 6 | 供货绑定删除保护 | countPurchaseInboundReferences 检查引用 | 保护链路完整，无法误删已被引用的绑定 |
| 7 | 过期扫描与库存脱钩 | markExpiredBatches 仅改批次状态，不联动 stock 汇总表 | 过期批次库存仍计入总库存，需通过报损手动处理 |

## 八、链路完整性结论

**已闭合的链路：**
- 供应商 → SKU 供货绑定 → 采购入库按供应商选货 → 校验绑定状态与最小采购量 ✅
- 采购入库明细 → 库存批次（1:1 创建，批次可反向追溯到入库明细） ✅
- 库存批次 → 库存汇总表（入库增加、出库扣减、报损扣减，均有流水） ✅
- 供货绑定删除保护（已被采购引用的绑定禁止删除） ✅
- 出库 FEFO 消费（按到期日排序，批次耗尽自动标记 DEPLETED） ✅

**尚存开口的链路：**
- 采购单无审批流，创建即终态，无法追溯审批过程
- 过期批次仅标记状态，不自动触发库存调整
- 操作人仅记录用户名，无用户 ID 快照
