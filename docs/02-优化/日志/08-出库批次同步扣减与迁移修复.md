# 08 - 出库批次同步扣减与迁移修复

> 记录时间：2026-06-01
> 项目阶段：优化阶段 - 完成出库批次同步扣减，并修复本地数据库迁移缺失导致的 500 异常

---

## 一、背景与目标

前两个阶段已经完成库存批次基础能力，并将采购入库的批次创建、批次流水写入收敛到 `StockService.increase(command)` 内部。本次继续补齐出库侧能力：出库不再只减少 SKU 总库存，而是同步扣减 `stock_batch` 的批次剩余数量，并写入 `stock_batch_log`。

核心目标：

- `StockService.decrease()` 仍作为出库库存扣减的统一入口。
- 出库时先锁定 `stock` 总库存记录，校验总库存是否充足。
- 按 FEFO 规则扣减批次库存：到期日期越早越先扣，同到期日按批次 ID 升序。
- 批次扣减后同步写入 `stock_log` 和 `stock_batch_log`。
- 批次剩余数量为 0 时更新批次状态，避免后续继续参与可用批次扣减。

---

## 二、调整后的核心流程

```text
OutboundService.create()
  ↓
写入 outbound_order 并取得 outbound_order.id
  ↓
StockService.decrease(skuId, baseQuantity, "OUTBOUND_ORDER", outboundId)
  ↓
锁定 stock 记录
  ↓
校验 stock.quantity 是否充足
  ↓
StockBatchService.consumeByFefo()
  ├─ 锁定可用 stock_batch
  ├─ 按 expire_date asc, id asc 分摊扣减数量
  └─ 更新 stock_batch.quantity 与 stock_batch.status
  ↓
更新 stock.quantity
  ↓
写入 stock_log
  ↓
写入 stock_batch_log
```

## 三、后端变更

### `StockService`

| 调整点 | 说明 |
|---|---|
| 保留 `decrease(Long skuId, int quantity)` | 兼容旧调用，内部使用默认来源 `STOCK_DECREASE` |
| 新增 `decrease(Long skuId, int quantity, String sourceType, Long sourceId)` | 支持调用方传入业务来源，用于批次流水追溯 |
| 出库扣减链路升级 | 总库存校验后调用 `StockBatchService.consumeByFefo()` |
| 批次流水写入 | 总账扣减成功后调用 `writeOutboundLogs(...)` 写出库批次流水 |

### `StockBatchService`

新增：

- `consumeByFefo(Long skuId, int quantity)`
- `writeOutboundLogs(List<BatchConsumption> consumptions, String sourceType, Long sourceId)`
- `BatchConsumption` 批次扣减结果记录

关键规则：

| 规则 | 说明 |
|---|---|
| 扣减顺序 | `expire_date asc, id asc` |
| 可扣批次 | `quantity > 0` 且 `status = 'AVAILABLE'` |
| 批次不足 | 抛出业务异常，不更新批次数量 |
| 批次扣完 | `status` 更新为 `DEPLETED` |
| 批次仍有剩余 | `status` 保持 `AVAILABLE` |
| 批次流水变化量 | 出库为负数，例如 `-5` |
| 批次流水来源 | 当前出库来源为 `OUTBOUND_ORDER` |

### `StockBatchMapper`

新增：

- `findConsumableBySkuIdForUpdate(Long skuId)`
- `updateRemainingQuantityAndStatus(Long batchId, Long skuId, int quantity, String status)`

批次锁定 SQL 以 FEFO 顺序查询并加 `for update`，保证同一 SKU 并发出库时不会重复扣减同一批次。

### `OutboundService` 与 `OutboundMapper`

| 调整点 | 说明 |
|---|---|
| `OutboundMapper.insert(...)` 返回自增 ID | 使用 `GeneratedKeyHolder` 取得 `outbound_order.id` |
| 出库扣减传入来源 | `stockService.decrease(..., "OUTBOUND_ORDER", outboundId)` |
| 插入 ID 为空 | 抛出“出库单保存失败”业务异常 |

## 四、数据库变更

### 批次状态字段

`stock_batch` 新增：

```sql
status VARCHAR(20) NOT NULL DEFAULT 'AVAILABLE'
```

状态取值：

| 状态 | 含义 |
|---|---|
| `AVAILABLE` | 批次仍有剩余数量，可参与出库扣减 |
| `DEPLETED` | 批次已扣完，不再参与出库扣减 |

### 迁移脚本

新增：

- `system/sql/09_add_stock_batch_status.sql`

脚本内容：

- 为已存在的 `stock_batch` 表补充 `status` 字段。
- 按现有 `quantity` 初始化状态：`quantity = 0` 为 `DEPLETED`，其余为 `AVAILABLE`。
- 增加状态枚举检查约束。

同时同步更新：

- `system/sql/08_add_stock_batch.sql`
- `system/sql/00_终版.sql`
- `system/sql/market.sql`

## 七、问题复盘：采购入库和出库返回 500

### 现象

调用采购入库和出库管理时，前端得到“系统内部异常”，接口响应为：

```json
{
  "code": 500,
  "message": "系统内部异常",
  "data": null
}
```

### 排查结论

后端代码已经开始读写 `stock_batch.status`：

- 采购入库创建批次时执行 `insert into stock_batch(..., status, ...)`
- 出库扣减批次时执行 `where status = 'AVAILABLE'`

但本地 `market.stock_batch` 实际表结构仍是旧版本，没有 `status` 列。因此进入批次链路时触发 SQL 异常，并被全局异常处理包装为 `code=500`。

### 修复动作

执行 `09_add_stock_batch_status.sql` 等价迁移后，本地数据库确认：

```text
has_status_column=true
AVAILABLE=3
DEPLETED=1
```

随后重新验证采购入库创建和出库创建均返回 `code=0`。

### 注意事项

本次排查过程中，为复现和验证创建了 1 条调试采购入库记录和 1 条调试出库记录。
