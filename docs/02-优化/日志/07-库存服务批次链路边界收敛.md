# 07 - 库存服务批次链路边界收敛

> 记录时间：2026-06-01
> 项目阶段：优化阶段 - 收敛采购入库、库存总账与库存批次之间的服务边界

---

## 一、背景与目标

上一阶段已经完成库存批次能力：采购入库每条明细会生成一条 `stock_batch`，并写入 `stock_batch_log`。当时的处理链路分散在 `PurchaseInboundService` 中：

```text
StockBatchService.createFromPurchaseInboundItem(...)
StockService.increase(...)
StockBatchService.writePurchaseInboundLog(...)
```

这会让采购入库服务同时承担单据、库存总账和批次流水三类职责。为了降低业务服务之间的耦合，并保证库存总账与批次记录的原子一致性，本次将批次创建和批次流水写入收敛到 `StockService.increase()` 内部。

目标：

- `PurchaseInboundService` 只负责创建采购入库单和采购入库明细。
- 每条采购明细保存后，只调用一次 `stockService.increase(command)`。
- `StockService` 统一完成 `stock`、`stock_log`、`stock_batch`、`stock_batch_log`。
- 避免出现 SKU 总库存增加成功，但批次或批次流水未生成的情况。

---

## 二、调整后的核心流程

采购入库创建时，每条明细按以下流程处理：

```text
写入 purchase_inbound_item
  ↓
组装 StockIncreaseCommand
  ↓
StockService.increase(command)
  ├─ 锁定并增加 stock 总库存
  ├─ 写入 stock_log
  ├─ 调用 StockBatchService 创建 stock_batch
  └─ 调用 StockBatchService 写 stock_batch_log
```

`PurchaseInboundService` 不再直接注入或调用 `StockBatchService`。

## 三、后端变更

### 新增命令对象

新增：

- `system/backend/src/main/java/com/supermarket/inventory/stock/dto/StockIncreaseCommand.java`

字段包括：

| 字段 | 说明 |
|---|---|
| `skuId` | SKU ID |
| `quantity` | 本次增加的基础单位数量 |
| `purchasePrice` | 采购入库单价快照 |
| `productionDate` | 生产日期 |
| `expiryDate` | 到期日期 |
| `purchaseInboundId` | 采购入库单 ID |
| `purchaseInboundItemId` | 采购入库明细 ID |
| `sourceType` | 来源类型，当前为 `PURCHASE_INBOUND_ITEM` |
| `sourceId` | 来源 ID，当前为采购入库明细 ID |

### `PurchaseInboundService` 调整

| 调整点 | 说明 |
|---|---|
| 移除 `StockBatchService` 依赖 | 采购服务不再直接创建批次或写批次流水 |
| 新增 `toStockIncreaseCommand(...)` | 将采购明细转换为库存增加命令 |
| 明细处理调用收敛 | 每条明细只调用 `stockService.increase(command)` |
| 到期日期计算前移 | 由 `productionDate.plusDays(shelfLifeDays)` 得出 `expiryDate` 并传给库存服务 |

### `StockService` 调整

| 调整点 | 说明 |
|---|---|
| 注入 `StockBatchService` | 库存服务成为总库存与批次写入的统一入口 |
| 新增 `increase(StockIncreaseCommand command)` | 专用于采购入库等需要批次记录的库存增加场景 |
| 保留旧 `increase(Long skuId, int quantity, String changeType)` | 普通入库仍可只增加 SKU 总库存并写 `stock_log` |
| 内部转换 `StockBatchCreateCommand` | 根据命令创建批次，`shelfLifeDays` 由 `expiryDate - productionDate` 计算 |
| 统一事务 | `increase(command)` 使用 `@Transactional`，覆盖总库存、总流水、批次、批次流水 |

### `StockBatchService` 调整

`writePurchaseInboundLog(batch)` 保留，兼容原有内部调用语义。

新增重载：

```java
writePurchaseInboundLog(StockBatch batch, String sourceType, Long sourceId)
```

该重载由 `StockService` 调用，用于将来源类型和来源 ID 从 `StockIncreaseCommand` 传递到 `stock_batch_log`。

---

## 四、事务一致性

本次调整后，采购入库主流程仍由 `PurchaseInboundService.create()` 开启事务；库存增加命令进入 `StockService.increase(command)` 后，在同一事务上下文内完成：

- 更新 `stock`
- 写入 `stock_log`
- 插入 `stock_batch`
- 写入 `stock_batch_log`

任一步抛出异常时，采购入库单、明细、总库存、总库存流水、批次和批次流水都会回滚。

## 七、后续可扩展方向

- 将后续按批次出库、批次盘点等能力继续收敛到库存服务边界内。
- 为 `StockIncreaseCommand.sourceType` 建立枚举或常量类，避免后续字符串散落。
- 当 `stock_log` 后续需要来源追踪时，可扩展来源字段，与 `stock_batch_log` 的来源模型保持一致。

---

*本次调整完成后，采购入库服务回归单据职责，库存服务成为库存总账和批次库存的统一写入入口，库存变更的一致性边界更清晰。*
